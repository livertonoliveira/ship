# ADR-001: Skill-wrapper + Named Agent Pattern

**Status:** Accepted  
**Date:** 2026-05-24  
**Issue:** MOB-1554

---

## Context

Ship's 17 SKILL.md files mix slash-command wrapper logic with implementation methodology in a single file. This creates two problems:

1. **Cognitive noise** — the LLM must parse dispatch logic and domain logic in the same context window.
2. **Structural ambiguity** — there is no clear boundary between "orchestrator" and "worker" roles.

The Claude Code plugin system supports a dedicated `agents/` directory within a plugin. Files placed there become named Agent definitions invocable via `subagent_type`. This enables a clean separation of concerns.

---

## Decision

Adopt the **Skill-wrapper + Named Agent** pattern for Ship skills that carry implementation methodology.

### File locations

| Artifact | Path |
|----------|------|
| Named Agent source | `src/agents/ship-<name>.md` (canonical) |
| Skill-wrapper source | `src/skills/<name>/SKILL.md` (canonical) |
| Build output | `plugins/ship/agents/ship-<name>.md` and `plugins/ship/skills/<name>/SKILL.md` (generated by `plugins/ship/scripts/build.js`, never edit by hand) |

> **Build pipeline note:** `src/` is the source tree that gets compiled/copied into `plugins/ship/`. Both locations must be kept in sync. When applying the pattern to a skill, update both `src/` and `plugins/ship/` simultaneously.

### Named Agent frontmatter

```yaml
---
name: ship-<name>
description: "<one-line description of what the worker does>"
tools: [Read, Glob, Grep, Bash, Edit, Write, Agent, mcp__linear-server__*]
model: sonnet
---
```

Include `Agent` in `tools` only when the worker spawns parallel sub-agents (e.g., for fan-out across independent modules). Remove it for single-flow workers.

The `prompt` field is the body of the file (everything after the frontmatter). It contains the full implementation methodology — no wrapper logic.

### Skill-wrapper structure (~50 lines)

```yaml
---
name: <name>
description: "<user-facing description>"
argument-hint: "<hint>"
allowed-tools: Read, Agent, mcp__linear-server__*
user-invocable: true
model: haiku
context: fork
---
```

The wrapper body does three things only:
1. Parse `$ARGUMENTS` to extract task ID and flags
2. Load minimal context from `ship/config.md` (storage mode, artifact language)
3. Invoke the Named Agent via the Agent tool with `subagent_type: ship-<name>`, passing all context inline

### When to create a Skill-wrapper

Only when the skill makes sense as a **standalone user command** (e.g., `/ship:develop`, `/ship:test`). Pure orchestration phases invoked only by `ship:run` do not need a Skill-wrapper — the orchestrator calls the Agent directly.

### How orchestrators pass context inline

Orchestrators (`run`, `audit:run`) call the Agent tool directly:

```
Agent tool call:
  subagent_type: ship-develop
  prompt: |
    Task: <task-id> — <title>
    Artifact language: pt-BR
    Scratch dir: .context/ship-run/<task-id>/

    ## Spec
    <inline task description and ACs>

    ## Design
    <inline design section relevant to this module>
```

This avoids N × file-size token cost when fanning out to multiple agents. The orchestrator reads shared artifacts once, slices them per agent, and passes slices inline.

---

## src/ → plugins/ship/ Build

`src/` is the single source of truth for both skills and agents. The build script at `plugins/ship/scripts/build.js` regenerates `plugins/ship/skills/` and `plugins/ship/agents/` from `src/skills/` and `src/agents/`, resolving any `@ship/<file>.md` reference inline. A reference with an anchor (`@ship/report-templates.md#finding-entry`) inlines only that markdown section; an anchorless reference inlines the whole file.

**Workflow:**
1. Edit `src/agents/ship-<name>.md` or `src/skills/<name>/SKILL.md`.
2. Run `cd plugins/ship && npm run build`.
3. Commit both `src/` and the regenerated `plugins/ship/` together.

Never edit `plugins/ship/agents/` or `plugins/ship/skills/` directly — the next build will overwrite the changes.

---

## Consequences

- Skill-wrapper files stay thin (~50 lines) — easy to audit and maintain.
- Implementation methodology lives in a dedicated Agent definition — clear ownership.
- Orchestrators gain explicit `subagent_type` invocations — intent is self-documenting.
- Future refactors (M4–M6) follow this same pattern for `test`, `perf`, `security`, `review`.
- `src/` is the single source of truth; `plugins/ship/` is build output, regenerated by `npm run build`. No more manual mirroring or silent drift.
